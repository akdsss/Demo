# 设计文档评审与优化建议

使用C#进行开发

## 文档状态更正

`generalGDDforAI.md` 与 `DevGuidelines.md` 已按 UTF-8 without BOM 保存，并通过字节检查确认包含正常中文内容。此前在工具输出中看到的乱码，是终端/控制台显示编码链路造成的误判，不应视为文档损坏。

后续开发应以 `generalGDDforAI.md` 作为 GDD 权威来源，以 `DevGuidelines.md` 作为 AI 修改约束与编码保护规范。

## 防止编码误判的流程

如果后续再次看到中文乱码，先按以下顺序核验：

1. 检查文件是否能以 UTF-8 解码。
2. 检查是否存在 UTF-8 BOM；项目建议使用 UTF-8 without BOM。
3. 检查原始字节中是否存在正常中文 UTF-8 字节，例如“战斗系统”“时间轴”等关键词。
4. 区分“文件内容损坏”和“终端显示乱码”。PowerShell 或工具输出显示乱码时，不直接判定文件损坏。
5. 只有在 UTF-8 解码失败、出现替换字符，或文件字节中确实不存在正常中文时，才报告编码事故。

这个流程应写入后续数据检查工具和 AI 工作习惯中，避免把正常文档误判为损坏文档。

## 高优先级问题

### 1. Demo 范围已经明确，但仍需要分期落地

更新后的 GDD 明确目标平台为 Windows，生成范围是完成文档所有要求，并做出包含 3 个关卡的可玩 demo。这比“原型验证”范围更大，已经包含完整战斗循环、3 个关卡、角色成长、百科、教程和资源接入。

建议把 3 关 demo 拆成阶段验收：

1. 教学关：验证时间轴、区域、基础技能、百科引导。
2. 小型敌方团体：验证普通敌人 AI、蓄力、远程/近战基础组合。
3. 双精英战：验证狂怒、支援敌人、范围远程、复杂 AI。
4. Boss 战当前只出现标题，若确实要纳入 demo，需要补充 Boss 的数据与胜负流程；若不纳入 3 关 demo，应明确它是后续扩展。

### 2. 美术与音频范围需要防止误解

GDD 明确：演出阶段每个指令约 3 秒的战斗播片不需要现在生成，先用占位资源搭建；其他缺少美术由 AI 生成，风格优先考虑水墨风；BGM 暂不考虑；攻击、受击等战斗相关音效也暂不考虑。

建议：

- 把“战斗播片占位”作为独立资源类型，不阻塞战斗内核开发。
- 先生成 UI 图标、区域图标、角色头像、敌人头像、百科图标等静态资源。
- 不制作攻击/受击 SFX，但保留 UI hover/click、阶段提示等轻量音效接口，因为 GDD 中仍提到这些反馈。
- 美术生成提示词、源图和导出图分目录保存，避免后续风格漂移。

### 3. 规则复杂度明显高于 Game Jam 常规可控范围

GDD 包含六时点时间轴、准备/演出双阶段、技能优先级、延迟伤害、范围伤害、区域修正、离开区域后的保留效果、状态触发、敌人决策、百科、教程、成长、技能解锁等。核心机制很有辨识度，但一次性完整实现风险较高。

建议 MVP 顺序：

1. 六时点时间轴与单角色行动放置。
2. 近战、远程、移动、防御、治疗五类基础技能。
3. 区域修正先实现被动数值类。
4. 再加入延迟伤害、范围伤害、状态触发。
5. 最后补敌人高级 AI、百科分页、成长解锁。

### 4. “演出阶段”和“结算阶段”边界需要明确

GDD 中演出阶段主要根据后台结算结果播放动画和战斗记录。但部分状态会在回合开始、回合结束、某时点、离开区域后生效，容易让实现把逻辑散落到动画播放流程里。

建议：

- 后台结算必须生成不可变的 `CombatEvent` 列表。
- 演出阶段只消费事件，不再次改变战斗结果。
- 所有状态触发都进入结算管线，不能在 UI 动画回调里改生命值或状态。

### 5. 区域规则很有特色，但缺少统一数据模型

八卦区域同时承担站位、地形修正、状态来源和百科内容。部分规则还涉及“离开区域后保留一次近战加成”“某状态在特定区域增强”等跨系统逻辑。

建议：

- 区域用 `AreaDefinition` 配置。
- 区域效果拆成 `Modifier`、`Trigger`、`StatusApplication` 三类。
- 离开区域后的保留效果不要写死在区域脚本里，应转化为临时状态。

### 6. 技能合法性检查需要拆成独立规则

GDD 中反复出现这些失败原因：目标不在同一区域、目标在同一区域、已处于目标区域、目标失去战斗能力、行动机会不足、MP 不足等。

建议：

- 做独立 `SkillValidator`。
- 失败原因用枚举与本地化文本映射。
- 敌人 AI 与玩家 UI 共用同一套合法性检查。

### 7. 敌人 AI 规则混合了行为表和特殊脚本

普通近战、普通远程、刺客、支援、Boss 等敌人有不同权重、释放前提、目标选择与特殊状态。若直接写在 C# 类中，后续会很难平衡。

建议：

- 基础行为用数据表驱动：技能、权重、前提、目标选择器。
- 特殊行为用少量策略类挂接。
- 所有 AI 输出都应是和玩家一样的 `PlannedAction`。

### 8. 数据保护要求需要从第一天落实

`DevGuidelines.md` 明确要求保护中文文本、配置数据、技能/怪物/区域/状态等核心数据，并禁止批量删除文件或目录。这些约束应视为开发流程的一部分，而不是后期清理事项。

建议：

- 所有文本与配置统一 UTF-8 without BOM。
- 数据脚本必须显式说明输入、输出、是否覆盖、编码策略。
- 批量修改配置前先做备份或确认版本控制状态。
- 不使用 `Remove-Item -Recurse`、`rm -rf`、`rmdir /s` 等批量删除命令。

## 中优先级优化

### UI 文字和战斗记录需提前规范

战斗记录需要显示头像、名称、行动、伤害、治疗、战败等。建议从一开始就做 `BattleLogEntry` 数据结构，不要直接拼接 UI 文本。

### 教程内容应该脚本化

GDD 中教程要求逐步展示文字、UI 变化和参考图。建议使用 `TutorialStep` 配置：高亮目标、提示文本、等待条件、允许输入范围。

### 百科系统适合复用配置

百科展示区域、状态、技能说明。建议百科内容不要重复维护，而是尽量从区域/状态/技能配置生成，只在需要长文解释时增加 `LoreText` 字段。

### 数值表需要版本化

角色生命、攻击、MP、行动次数、技能消耗、敌人血量等很多。建议所有数值先落 JSON/CSV，不散落在场景和脚本中。

### 规则术语需要建立统一词表

GDD 中存在大量专有术语，例如时间点、时间轴、准备阶段、演出阶段、区域修正、刻印、闪避、燃烧、蓄力、狂怒等。建议在数据或文档中维护术语表，避免 UI、配置、代码注释和百科之间出现不同叫法。

### 玩家操作约定应进入输入映射

GDD 明确左键为确定、右键为返回。建议从一开始建立 Godot InputMap：`confirm`、`back`、`place_action_hold`、`inspect` 等，不要在 UI 代码中散落鼠标按键判断。

## 结论

这个设计的核心亮点是“在时间轴上预排指令，再看战斗像机关一样执行”。它适合做成小而深的策略游戏。最大风险不是机制不好，而是规则太多、状态触发太分散、数据没有结构化。开发时应先搭一个稳定结算内核，再让 UI、美术、敌人和关卡围绕它并行推进。

编码方面，本次误会说明：文件内容、保存编码、终端显示是三件不同的事。后续判断中文文档是否异常时，应以 UTF-8 字节检查和编辑器实际显示为准，不以单次终端输出为准。
