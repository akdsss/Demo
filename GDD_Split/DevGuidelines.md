# AGENTS.md

本文件用于约束 AI 在本项目中的开发行为。项目包含大量中文文本、战斗数值、技能配置、怪物行动逻辑、百科条目、教程文案和表格型数据。任何自动化修改都必须优先保护数据完整性，尤其避免中文文本和批量配置文件因脚本、编辑器或 Shell 默认编码差异而被悄悄破坏。

（此外，开发时使用语言为使用语言：C#）

## 一、数据与文本编码规则

1. 所有文本配置文件必须使用 UTF-8 读写，包括但不限于：
   - JSON
   - CSV
   - TSV
   - TXT
   - YAML / YML
   - Markdown
   - 本地化文本表
   - 百科条目数据
   - 技能、怪物、关卡、区域修正、状态效果等配置数据
2. 脚本写文件时，统一使用 UTF-8 without BOM。
3. 读写文本文件时，必须显式指定编码，不要依赖 PowerShell、CMD、Node、Python 或编辑器的默认编码。
4. 不要使用会强制保存为 ANSI、GBK、GB2312、Big5 等旧编码的编辑器或脚本。
5. 不要把中文内容从 Word、网页、聊天窗口或表格软件中直接复制进配置文件后立刻批量提交。必须先运行编码与结构检查。
6. 不要在未确认编码的情况下使用重定向、管道或 Shell 文本命令重写配置文件，例如：
   - `>` / `>>`
   - `Out-File` 未指定编码
   - `Set-Content` 未指定编码
   - `Add-Content` 未指定编码
   - `Export-Csv` 未指定编码
7. 如果必须用 PowerShell 写文本文件，应显式使用 UTF-8：
   - PowerShell 7：优先使用 `-Encoding utf8`
   - Windows PowerShell 5.1：确认是否会写入 BOM；若项目要求无 BOM，应使用专门脚本或 .NET UTF8Encoding(false)

## 二、重点保护的数据类型

以下数据是项目核心资产，AI 修改时必须格外谨慎：

1. 战斗系统数据
   - 六时点时间轴配置
   - 指令优先级
   - 行动次数
   - 准备阶段与演出阶段流程
   - 战斗记录显示文本
2. 区域与状态数据
   - 阳、阴、乾、兑、离、震、巽、坎、艮、坤
   - 区域修正名称与效果
   - 闪避、刻印、护盾、燃烧、罡风等状态效果
   - 持续时点、触发时机、离开区域后的保留规则
3. 技能数据
   - 技能名称
   - 技能类型与标签：近战、远程、移动、防御、特殊、单体、范围、治疗
   - 优先级
   - MP 消耗
   - 目标类型
   - 效果描述
   - 释放失败文本
   - 解锁关卡或升级说明
4. 怪物与关卡数据
   - 怪物生命值、攻击力、行动次数
   - 怪物技能列表
   - 释放前提
   - 目标选择逻辑
   - 随机权重
   - 狂怒、蓄力、延迟范围伤害等特殊逻辑
5. UI 与教程文本
   - 教学关展示文本
   - 百科条目
   - 按钮文字
   - 战斗提示
   - 阶段标题
   - 失败原因文案
6. 角色成长数据
   - 初始属性
   - 关卡胜利后的属性提升
   - 新技能解锁与技能升级

这些数据包含大量中文标点、全角符号、百分比、括号说明和专有名词。不要把它们当作普通字符串随意批量替换。

## 三、批量修改规则

1. 涉及大量数据修改前，必须先确认项目内是否已有专用工具或脚本，例如：
   - `Tools/Json`
   - `Tools/Data`
   - `Tools/Localization`
   - `Scripts/Data`
   - `Editor` 数据导入工具
2. 如果已有专用脚本，优先使用项目脚本，不要临时手写一次性批处理。
3. 批量修改前必须先备份目标文件，或确认 git 中已有干净可回退版本。
4. 批量修改后必须进行最少三类检查：
   - 编码检查：确认仍为 UTF-8 without BOM
   - 结构检查：JSON 可解析、CSV/TSV 列数一致、YAML 可解析
   - 语义抽查：抽查中文文本、技能效果、区域修正、怪物释放逻辑是否被误改
5. 不要对整个数据目录运行未经审查的格式化工具。
6. 不要为了“统一格式”重排所有 JSON 字段，除非项目已有明确排序规则。
7. 不要在一次提交中同时做数据迁移、字段重命名、文本润色和数值平衡。应拆分成可审查的小步骤。
8. 如果发现某个文件已有乱码，不要继续在该文件上叠加修改。应先停止，说明问题，并从备份或 git 历史恢复。

## 四、JSON / CSV / TSV 规则

### JSON

1. 修改 JSON 时必须使用 JSON 解析器读写，不要用正则或字符串替换直接改结构。
2. 写出 JSON 时必须保持 UTF-8 without BOM。
3. 不要丢失字段顺序，除非项目明确允许排序。
4. 不要把数字写成字符串，也不要把字符串 ID 自动转换成数字。
5. 不要删除未知字段。未知字段可能是引擎、编辑器或策划工具使用的扩展字段。
6. 对技能、怪物、关卡、百科、区域修正等 JSON，修改后必须至少验证：
   - 文件可解析
   - 必填字段存在
   - ID 唯一
   - 引用的技能、状态、区域、角色、资源路径存在
   - 中文字段没有乱码

### CSV / TSV

1. 修改 CSV / TSV 时必须使用可靠的表格解析库，不要用简单的 `split(',')` 或 `split('\t')` 处理复杂内容。
2. 中文描述中可能包含逗号、顿号、括号、百分号、引号和换行。不要用粗暴分割破坏字段。
3. 保持表头不变，尤其不要让 UTF-8 BOM 混入第一列表头。
4. 修改后必须检查每行列数一致。
5. 如果表格由 Excel、WPS 或 Google Sheets 导出，导入项目前必须确认编码是 UTF-8。

## 五、中文文本保护规则

1. 不要自动改写中文文案，除非用户明确要求润色。
2. 不要把中文标点替换成英文标点，除非项目已有统一规范。
3. 不要删除括号中的规则解释，例如：
   - `（刻印：克制闪避）`
   - `（燃烧：回合结束时，受到最大生命值6%的伤害。）`
4. 不要改变关键术语：
   - 时点
   - 时间轴
   - 准备阶段
   - 演出阶段
   - 区域修正
   - 刻印
   - 闪避
   - 罡风
   - 蓄力
   - 狂怒
   - 至阴
   - 至阳
5. 不要把中文技能名、区域名、状态名翻译成英文 ID，除非另有字段专门存放英文 ID。
6. 如果需要新增 ID，推荐使用稳定英文 ID；但显示文本仍保留中文字段。
7. 如果发现文本出现类似 `閫犳垚`、`锟斤拷`、`Ã`、`�` 的内容，应视为编码事故，停止继续修改并报告。

## 六、脚本约定

1. 新增数据处理脚本时，应放在项目约定的工具目录中，例如 `Tools/Json` 或 `Tools/Data`。
2. 脚本必须在文件开头说明：
   - 输入文件
   - 输出文件
   - 是否会覆盖原文件
   - 是否会创建备份
   - 编码策略
3. 脚本默认不得原地覆盖重要数据文件。若必须覆盖，先创建明确路径的备份文件。
4. 备份文件名应包含时间戳，例如：
   - `cards.json.bak.20260520-013000`
5. 脚本写出文本时必须显式指定 UTF-8 without BOM。
6. 脚本遇到解析失败、字段缺失、重复 ID、编码异常时，应停止执行，不要继续写出文件。
7. 脚本输出日志应说明修改了哪些文件、多少条记录、哪些字段。
8. 不要让脚本静默修复大量数据。自动修复必须列出修复摘要。

## 七、推荐检查脚本

如果项目中尚未存在类似工具，建议建立以下脚本，并在批量修改后执行：

1. `Tools/Json/JsonEncodingGuard.ps1`
   - 检查 JSON 是否为 UTF-8 without BOM
   - 检查 JSON 是否能解析
   - 抽查中文字段是否包含常见乱码特征
2. `Tools/Json/DataSchemaGuard.ps1`
   - 检查技能、怪物、区域、状态、百科等数据的必填字段
   - 检查 ID 唯一性
   - 检查引用关系
3. `Tools/Text/LocalizationEncodingGuard.ps1`
   - 检查中文本地化文本、教程文本、百科文本是否存在乱码
   - 检查是否混入异常控制字符
4. `Tools/Data/CsvTsvGuard.ps1`
   - 检查 CSV / TSV 编码
   - 检查表头
   - 检查列数一致

## 八、推荐工作流

1. 修改前：
   - 阅读相关 GDD 段落
   - 找到对应数据文件
   - 确认编码
   - 确认是否已有专用修改脚本
2. 修改中：
   - 使用解析器修改结构化数据
   - 显式指定 UTF-8 without BOM 写出
   - 保留未知字段
   - 保留中文原文和关键术语
3. 修改后：
   - 运行编码检查
   - 运行结构检查
   - 运行项目数据加载或最小启动测试
   - 抽查关键中文文本
4. 如果检查失败：
   - 不要继续追加修改
   - 不要反复运行修复脚本覆盖原文件
   - 优先从备份或 git 历史恢复
   - 再重新应用经过确认的脚本

## 九、AI 开发行为限制

1. AI 不得在未说明影响范围的情况下批量重写数据文件。
2. AI 不得凭记忆改动 GDD 中的规则。涉及战斗、技能、怪物 AI、区域修正、状态效果时，必须先查阅当前项目文档或数据。
3. AI 不得把“看起来重复”的中文条目自动合并。重复可能是不同关卡、不同角色或不同上下文的刻意设计。
4. AI 不得删除暂时未使用的数据字段。未使用字段可能是后续关卡、编辑器、调试工具或存档兼容所需。
5. AI 不得把全量配置文件重新格式化为另一种风格，除非用户明确要求。
6. AI 不得把 DOCX / GDD 中的正文直接作为最终配置文件写入。必须经过结构化整理、字段校验和编码检查。
7. AI 不得在未确认业务含义的情况下修改以下规则：
   - 近战与移动技能的同区域判定
   - 延迟范围伤害跨回合触发
   - 区域修正离开区域后的保留次数
   - 狂怒状态下的行动次数与释放方案
   - 技能优先级
   - 状态持续时点
   - 怪物目标选择逻辑
8. AI 如发现 GDD、数据文件、代码实现三者不一致，应先报告差异，不要擅自选择其中一个覆盖另外两个。

## 十、删除与清理限制

禁止批量删除文件或目录。

不要使用：

- `del /s`
- `rd /s`
- `rmdir /s`
- `Remove-Item -Recurse`
- `rm -rf`

需要删除文件时，只能一次删除一个明确路径的文件。

正确示例：

```powershell
Remove-Item "C:\path\to\file.txt"
```

如果需要批量删除文件，应停止操作，并询问用户，让用户手动删除或明确授权具体删除清单。

## 十一、出现异常时的处理

如果遇到以下情况，AI 必须停止继续写入并向用户说明：

1. 文件编码无法确认。
2. JSON / YAML 解析失败。
3. CSV / TSV 列数不一致。
4. 中文字段出现疑似乱码。
5. 同一个 ID 在多个数据文件中冲突。
6. 技能、状态、区域、怪物或资源路径引用不存在。
7. 批量脚本会影响超过一个核心数据文件，但没有备份方案。
8. GDD 规则与当前数据或代码实现冲突。

优先恢复数据正确性，再继续功能开发。
